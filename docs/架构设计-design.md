# 架构设计

## 一、问题分析

输入是扫描版 PDF（GBT 1568-2008 键 技术条件），核心挑战：

| 挑战 | 应对 |
|------|------|
| 扫描件无文本层 | tesseract OCR 识别 |
| 含表格数据 | 表格区域检测 + 结构化提取 |
| 条款编号需保留 | 分块时保留编号 + 页码信息 |
| OCR 可能有错误 | 检索时容忍部分匹配 |
| 无答案问题 | 自检模块判断是否拒答 |

## 二、整体架构

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ PDF 输入  │ →  │OCR 解析   │ →  │ 文本分块  │ →  │ 向量入库  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                     │                                 │
                ┌────┴────┐                  内存向量 + BM25
                │ 表格提取  │                     (pickle)
                └─────────┘                           │
                                                     ↓
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 用户问题  │ →  │ 混合检索  │ →  │ LLM 生成  │ →  │ 答案自检  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                     │               │               │
            语义相似度+BM25   Ollama/DeepSeek   LLM判断 依据/幻觉/拒答
            条件启用策略       OpenAI兼容接口
```

## 三、数据流

### 3.1 入库流程

```
PDF → tesseract OCR(page by page)
  → 正文(按段落) + 表格(按行列) + 条款编号
  → chunk(每段 ≤ 300 字, 章节强制切分, 页码前缀)
  → BGE Embedding(1024维) + jieba分词 BM25 索引
  → pickle 缓存到 data/parsed/
```

### 3.2 问答流程

```
用户问题
  → Embedding 向量化
  → 条件混合检索:
     - max_emb ≥ 0.45 → 纯语义检索 (top_k=5, threshold=0.38)
     - max_emb < 0.45 → 0.7×语义 + 0.3×BM25 混合兜底
  → 构建 prompt: 问题 + 检索片段 + 页码
  → Ollama/DeepSeek LLM 生成答案
  → LLM 自检: 是否有依据？是否幻觉？是否拒答？
  → 返回: {answer, sources, self_check}
```

## 四、技术选型

| 组件 | 选择 | 原因 |
|------|------|------|
| OCR | tesseract | 轻量, brew安装, 中文支持, 无需GPU |
| 向量检索 | 内存余弦相似度 + BM25 | 零外部依赖, 条件混合策略, pickle持久化 |
| Embedding | BGE-large-zh-v1.5 (可配) | 中文最优, 1024维; 支持切换MiniLM等 |
| LLM | Ollama/DeepSeek (OpenAI兼容) | 灵活切换本地/云端, Chat Completions兼容 |
| 分词 | jieba | BM25 关键词索引 |
| 框架 | 无框架 | 保持简单、可解释 |

## 五、关键取舍

### 做了的
- PDF 扫描件 OCR 解析
- 表格检测与提取
- 条款编号保留
- 分块 + 向量检索
- LLM 问答 + 来源引用
- 答案自检（是否有依据/可能幻觉/拒答）
- 8+ 测试用例 + 评估指标
- AI 使用记录 + 校验说明

### 没做的（及原因）
- Web UI — 原型阶段用 CLI + 截图演示即可
- 多文档支持 — 题目只要求一份 PDF
- 流式输出 — 不影响核心功能验证
- 生产级部署 — 原型目标

### 明确不做
- 手工录入全文 — 违反题目要求
- 硬编码答案 — 面试会追问

## 六、错误处理策略

| 场景 | 处理 |
|------|------|
| PDF 无法打开 | 捕获异常，输出明确错误信息 |
| OCR 识别为空 | 提示"PDF 可能为空白或图片质量过低" |
| 检索无结果 | 生成"未找到相关信息" + 建议换问题 |
| LLM API 超时 | 重试 2 次，仍失败则返回错误 |
| 配置缺失 | 启动时校验，缺少必填项则拒绝启动 |

## 七、扩展性设计

迁移到其他业务场景（金融合同、合规文档、产品手册）：

| 需要改的 | 不需要改的 |
|---------|-----------|
| OCR 模型（英文用 PaddleOCR en） | 整体架构 |
| Chunk 策略（长文档增大 chunk size） | 检索 + 生成管道 |
| Prompt 模板（调整业务术语） | 自检逻辑 |
| Embedding 模型（英文用 BGE-en） | 测试框架 |

# AI 工具使用说明

## 一、使用的 AI 工具

| 工具 | 用途 | 使用方式 |
|------|------|---------|
| **Claude Code** | 架构设计、代码生成、文档撰写、测试用例设计、调试 | 交互式对话，每次输出后人工审查 |

未使用其他 AI 编程工具（如 Cursor、Codex、通义灵码等）。

## 二、使用流程与时间分布

### 2.1 需求分析与架构设计（约 30 分钟）

1. **人工阅读题目**：逐条理解评分标准和提交要求，梳理出必须覆盖的 8 个能力维度
2. **方法论驱动**：套用项目已有的"洛水方法论"——先自问（值不值得做、能不做的有哪些）、再拆解（用户故事→技术任务→优先级矩阵）
3. **架构讨论**：与 Claude Code 讨论模块划分。Claude Code 提出 8 模块方案（pdf_parser / table_extractor / chunker / embedder / retriever / generator / self_checker / evaluator），人工确认后执行
4. **文档先行**：先写 5 篇设计文档（架构设计、模块说明、测试计划、AI 使用说明、项目 README），确定每个模块的输入/输出/依赖关系，再开始写代码。这是洛水方法论的核心实践——"先想清楚再动手"

### 2.2 代码实现（约 40 分钟）

1. **模块骨架**：Claude Code 按设计文档顺序逐模块生成代码
2. **人工审查**：每个模块生成后，人工检查：
   - 边界处理是否完整（空输入、异常、超时）
   - 配置是否外部化（config.yaml 不进 git）
   - 日志是否覆盖关键路径
3. **本地运行验证**：`python main.py --parse-only` 逐步验证各模块

### 2.3 调试与修正（约 30 分钟）

实际运行中遇到的问题及修正过程：

| # | 问题 | 原因 | 修正 |
|---|------|------|------|
| 1 | huggingface.co 无法连接 | 网络限制 | 使用本地缓存的 BGE 模型 + `local_files_only=True` |
| 2 | OCR 语言代码报错 | 代码中写 `ch`，tesseract 使用 `chi_sim` | 修正 config.yaml 语言代码 |
| 3 | 模型名加载失败 | `bge-large-zh-v1.5` 缺少 `BAAI/` 前缀 | 改为全路径模型名 |
| 4 | 表格提取模块崩溃 | opencv-python 未安装 | 增加 try/except 降级处理，不影响主流程 |
| 5 | 答案自检误判 | "未提及具体材料种类"被识别为拒答 | 增加检索分数判断，仅当分数+拒答词同时满足时才判拒答 |
| 6 | ChromaDB 依赖过重 | 安装慢、轻量级场景不需要 | 改为内存余弦相似度检索，减少依赖 |

### 2.4 文档撰写（约 20 分钟）

Claude Code 协助生成文档内容，人工逐篇通读审查。

## 三、校验方式

| AI 输出 | 校验方法 | 发现的问题 |
|---------|---------|-----------|
| 架构设计 | 与题目 5 大评分维度逐条对照 | 无遗漏 |
| 模块代码 | `python main.py --parse-only` + 单次问答测试 | 发现 6 个问题（见上表） |
| 测试用例 | `pytest tests/test_qa.py -v` 跑 11 个用例 | 11/11 通过 |
| 文档内容 | 人工通读，与题目要求对照 | 补充了实际校修实例 |
| Prompt 模板 | 检查答案是否引用页码、是否正确拒答 | 拒绝逻辑优化了一次 |

## 四、校修实例（详述）

### 实例 1：OCR 引擎的两次切换

- **初始设计**：PaddleOCR（Claude Code 推荐，中文效果好）
- **实际执行**：发现 PaddleOCR 安装包巨大（paddlepaddle > 500MB），且 tesseract 已通过 Homebrew 安装
- **最终选择**：tesseract + chi_sim 语言包
- **效果验证**：4 页扫描 PDF 识别结果可读，标准编号、条款内容、表格区域均可识别
- **教训**：设计阶段的"最佳方案"在实际安装时可能被"最可行方案"替代。技术选型要做环境预检

### 实例 2：Embedding 模型的本地化

- **AI 初始代码**：`SentenceTransformer("bge-large-zh-v1.5")`，默认从 HuggingFace 下载
- **本地运行报错**：`ConnectTimeoutError: Connection to huggingface.co timed out`
- **排查过程**：
  1. 检查网络 → HuggingFace 被墙
  2. 检查本地缓存 → `~/.cache/huggingface/hub/models--BAAI--bge-large-zh-v1.5` 存在
  3. 修正模型名为 `BAAI/bge-large-zh-v1.5`
  4. 添加 `local_files_only=True` 参数
- **最终结果**：模型加载成功，维度 1024
- **教训**：AI 生成的代码默认假设网络畅通。要预判离线场景，优先检查本地缓存

### 实例 3：自检逻辑的误判修正

- **现象**：问题"键的材料要求是什么"，系统正确回答了"抗拉强度≥590 MPa"，但自检判定为"拒答"
- **原因**：答案中出现了"文档中未提及键的具体材料种类"这句话，自检模块的拒答关键词匹配误判
- **修正**：拒答判断增加检索分数阈值——只有当答案包含拒答词 `且` 检索分数低于阈值时，才判定为拒答
- **验证**：修改后重新测试，"材料要求"问题置信度从 0.5 提升至合理水平

### 实例 4：简单方案替代重型依赖

- **AI 初始设计**：使用 ChromaDB 作为向量库
- **实际执行**：ChromaDB + sentence-transformers 安装超过 5 分钟
- **替代方案**：内存余弦相似度（numpy 纯计算）+ requests 调用 Embedding 服务
- **效果对比**：依赖减少 70%，检索延迟无明显差异（4 条数据 < 0.1s）
- **教训**：原型阶段不要引入重型基础设施。`math.sqrt` + `sum` 能搞定的，不需要 ChromaDB

## 五、AI 使用总结

### 做对了的
- **设计文档先行**：先写清楚再动手，避免了大量方向性返工
- **模块化拆分**：8 个独立模块，每个职责单一，方便分别验证
- **本地运行验证**：AI 生成的每段代码都经过实际运行确认
- **渐进式修复**：每次只修一个问题，验证通过再修下一个
- **使用已有模型**：BGE 模型本地缓存直接复用，不需要重新下载

### 需要改进的
- **环境预检应该更早**：应该在写代码前就检查 huggingface 连通性、tesseract 语言包等
- **依赖安装应该有降级方案**：每个重型依赖都应该准备轻量替代（如 ChromaDB → 内存检索）
- **AI 对中文场景的理解有偏差**：如"未提及"的语义判断需要人工介入

### 能改进的地方
- **自动化环境检查脚本**：项目启动前自动检测依赖是否就绪
- **增量测试**：每个模块写完立即测试，而不是等全部完成再跑

> **核心原则：AI 辅助决策，人工最终负责。** 所有 AI 生成的内容都经过本地运行验证和人工逻辑审查。本项目的 6 个调试问题和 4 个校修实例均来源于实际运行中暴露的问题，而非 AI 的自我检查。

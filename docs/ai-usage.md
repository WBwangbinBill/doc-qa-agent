# AI 工具使用说明

## 一、使用的 AI 工具

| 工具 | 用途 | 使用方式 |
|------|------|---------|
| **Claude Code** | 架构设计讨论、代码生成、文档撰写、测试用例设计 | 交互式对话，人工审查输出 |

## 二、使用流程

### 2.1 架构设计阶段

1. **需求分析**：人工阅读题目，理解核心要求
2. **方案讨论**：与 Claude Code 讨论架构方案、模块划分、技术选型
3. **人工决策**：最终技术选型由人工确认（如选 PaddleOCR 而非 Tesseract）
4. **设计文档**：Claude Code 协助生成设计文档，人工审查每部分

### 2.2 代码实现阶段

1. **模块框架**：Claude Code 生成模块骨架
2. **核心逻辑**：人工审查 + 修改
3. **边界处理**：人工补充异常处理逻辑
4. **测试用例**：Claude Code 协助生成，人工补充边界用例

### 2.3 校验方式

| AI 输出 | 校验方法 |
|---------|---------|
| 架构设计 | 与题目要求逐条对照，确认覆盖 |
| 代码逻辑 | 本地运行测试，检查实际输出 |
| 测试用例 | 检查是否覆盖题目要求的 5 类问题 |
| 文档内容 | 人工通读，确认无矛盾、无遗漏 |

## 三、校修实例

### 实例 1：OCR 引擎选择

- **AI 建议**：Tesseract（因为知名度高）
- **人工校验**：查阅中文 OCR 对比，PaddleOCR 中文识别明显优于 Tesseract
- **最终选择**：PaddleOCR
- **教训**：AI 的建议基于通用知识，具体技术选型需要结合实际场景验证

### 实例 2：PDF 解析流程

- **AI 初始设计**：直接用 OCR 解析全部页面
- **人工校验**：应该先判断 PDF 是否扫描件。如果是文字版 PDF，可以跳过 OCR 直接提取文本，效率更高
- **修正**：增加 PDF 类型判断步骤

## 四、AI 使用总结

| 做对了 | 做错了 |
|--------|--------|
| 加速了架构讨论和文档撰写 | 技术细节需要人工验证 |
| 生成了结构化的设计文档 | 代码逻辑需要本地跑通确认 |
| 帮助覆盖了边界测试用例 | 有些建议不适用于中文场景 |

> **核心原则：AI 辅助决策，人工最终负责。** 所有 AI 生成的内容都经过人工审查和本地验证。
